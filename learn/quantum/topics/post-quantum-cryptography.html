1) The quantum threat to public-key crypto

Shor's algorithm solves integer factoring and discrete logarithms in polynomial time on a quantum computer, breaking RSA, Diffie–Hellman, and elliptic-curve cryptography (ECC) that secure today's internet, banking, and government communications.

  • Estimated timeline for cryptographically relevant quantum computers (CRQC): uncertain, but "harvest now, decrypt later" attacks motivate immediate migration.
  • Symmetric cryptography (AES, SHA-256) is less affected — Grover's algorithm effectively halves key strength.
  • Post-quantum cryptography (PQC) = classical algorithms believed secure against quantum attacks.
  • NIST finalised first PQC standards in 2024 (FIPS 203, 204, 205).
Key idea: PQC replaces vulnerable public-key primitives with ones based on problems that no known quantum algorithm solves efficiently.

Prerequisites: Shor's Factoring Algorithm, Cryptography.

Additional candidates under evaluation: FALCON (lattice signatures, compact), BIKE and HQC (code-based KEMs). NIST plans a second round of standardisation for backup algorithms.

3) Lattice-based cryptography

Learning With Errors (LWE): Given pairs \((\mathbf{a}, b = \langle\mathbf{a}, \mathbf{s}\rangle + e \bmod q)\) where \(\mathbf{s}\) is a secret vector and \(e\) is small noise, recovering \(\mathbf{s}\) is hard. The search-LWE problem connects to worst-case lattice problems (\(\gamma\)-SVP).

Module-LWE (used in Kyber/Dilithium) works over module lattices, balancing security and performance. Kyber-768 (NIST Level 3) has public key ~1.2 KB and ciphertext ~1.1 KB.

Kyber KEM flow: Alice generates key pair \((pk, sk)\); Bob encapsulates a shared secret \(K\) into ciphertext \(c = \text{Encaps}(pk)\); Alice decapsulates \(K = \text{Decaps}(c, sk)\). Based on IND-CCA security from a CPA-secure LWE construction plus Fujisaki–Okamoto transform.

Mathematical background: Matrices, Group Theory.

4) Hash-based signatures (SPHINCS+)

Hash-based signatures rely only on the security of cryptographic hash functions — conservative assumptions with no known quantum weakness beyond Grover's quadratic speedup (countered by doubling output length).

SPHINCS+ builds a hypertree of Merkle trees: a few long-term keys sign many one-time signature (OTS) keys, which sign individual messages. Security reduction:

\(\text{EUF-CMA}_{\text{SPHINCS+}} \leq \text{PRF}_{H} + (h+1) \cdot \text{EUF-CMA}_{\text{OTS}}\)

Advantages

  • Minimal security assumptions
  • Well-understood analysis
  • NIST backup standard (FIPS 205)

Drawbacks

  • Large signatures (~8–50 KB)
  • Slower signing than lattice schemes
  • Stateful variants (XMSS) need careful key management

Lesson from broken schemes: Rainbow and SIKE were broken by classical attacks after NIST shortlisting. PQC migration should include crypto-agility — ability to swap algorithms without infrastructure redesign.

6) Grover's impact on symmetric crypto

Grover's algorithm searches an unstructured database of \(N\) items in \(O(\sqrt{N})\) queries, effectively halving the security bits of symmetric-key algorithms:

  • AES-128 → ~64-bit quantum security. Migrate to AES-256 for 128-bit quantum security.
  • SHA-256 → ~128-bit quantum collision resistance. Use SHA-384/SHA-3 for long-term security.
  • HMAC and KDFs: increase output length proportionally.

Related: Grover's Search Algorithm.

7) Migration strategy

  1. Inventory: catalogue all uses of RSA, DH, ECDH, ECDSA in systems and protocols.
  2. Prioritise: long-lived secrets (TLS certificates, document signing, firmware) first.
  3. Hybrid mode: combine classical + PQC key exchange (e.g. X25519 + Kyber768) during transition.
  4. Test: PQC increases key/ciphertext sizes — test bandwidth, latency, certificate chains.
  5. Crypto-agility: design protocols to accept multiple algorithm identifiers.
Standards bodies: IETF (hybrid TLS in RFC drafts), NSA CNSA 2.0 (mandates PQC for national security systems by 2035), BSI (German guidance), NCSC (UK).

QKD is not a drop-in PQC replacement — see Quantum Key Distribution for the distinction.

8) Implementation considerations

Performance

  • Kyber encaps/decaps: ~100 μs (software)
  • Dilithium sign/verify: ~1 ms / ~300 μs
  • SPHINCS+ sign: ~100 ms (large signatures)

Side channels

  • Constant-time implementations required
  • Lattice schemes vulnerable to timing attacks if not careful
  • Hardware security modules (HSMs) adding PQC support

Libraries: liboqs (Open Quantum Safe), BoringSSL/OpenSSL PQC branches, PQClean reference implementations. Cloud providers (Google, Cloudflare, AWS) have deployed hybrid Kyber in TLS experiments.
