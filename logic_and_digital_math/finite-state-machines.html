Finite State Machines

Digital logic guide. Moore and Mealy models, state diagrams, and a step-by-step design example.

← Pure Math Subjects Catalog

1) What is a finite state machine?

A finite state machine (FSM) is a sequential model with a finite set of states, transitions between states driven by inputs, and outputs that depend on state (and possibly inputs).

  • State register: flip-flops holding the current state encoding
  • Next-state logic: combinational circuit computing the next state from current state and inputs
  • Output logic: combinational circuit producing outputs
Everywhere in digital design: Traffic-light controllers, UART receivers, CPU control units, vending machines, and protocol handlers are all FSMs.

Prerequisite: Sequential Logic, Combinational Logic.

2) Moore vs Mealy machines

Moore machine

Outputs depend only on the current state. Outputs change synchronously with state, one clock after a transition completes.

Mealy machine

Outputs depend on current state and inputs. Outputs can change immediately when inputs change, even mid-cycle.

PropertyMooreMealy
Output function\(Z = g(S)\)\(Z = g(S, X)\)
Typical state countOften more statesOften fewer states
Output timingAligned to clock edgesCan be asynchronous to clock
Glitch sensitivityLowerHigher (input glitches affect output)
Rule of thumb: Use Moore when clean, clock-aligned outputs matter. Use Mealy when minimising state count or reacting immediately to inputs is important.

3) State diagrams

A state diagram (state transition graph) is the primary design notation for FSMs:

  • Circles: states, labelled S0, S1, … or with meaningful names (IDLE, WAIT, SEND)
  • Arrows: transitions, labelled with input condition / output action
  • Moore convention: outputs written inside the state circle: S0 / Z=0
  • Mealy convention: outputs on the transition arrow: input / output
        ┌─────────┐
   ───│  IDLE   │─── reset
   │  │  Z = 0  │   │
   │  └────┬────┘   │
   │       │ start=1
   │       ▼
   │  ┌─────────┐
   └──│  BUSY   │
      │  Z = 1  │
      └─────────┘

Always mark the initial state (usually after reset) and ensure every state has defined transitions for all relevant input combinations (or a default / error state).

4) State assignment and encoding

Each state must be encoded as a bit pattern stored in flip-flops. Common schemes:

  • Binary encoding: \(k\) flip-flops encode up to \(2^k\) states; minimises register size
  • One-hot encoding: one flip-flop per state; only one bit is 1 at a time; simplifies next-state logic in FPGAs
  • Gray code encoding: adjacent states differ by one bit; reduces power and hazard risk during transitions
FPGA tip: One-hot encoding often maps well to flip-flop-rich architectures even though it uses more registers than binary.

5) Design procedure

  1. Define I/O: list all inputs, outputs, and their meaning
  2. Draw state diagram: identify states and all transitions
  3. Build state table: rows = current state; columns = input combinations; cells = next state and output
  4. Choose state encoding: binary, one-hot, or Gray
  5. Derive Boolean equations for next-state and output logic (K-maps or HDL)
  6. Implement: state register (D flip-flops) + combinational next-state and output blocks
  7. Simulate: verify with test vectors and timing diagrams

6) Design example: sequence detector "101"

Detect the bit sequence 1-0-1 in a serial input stream. Output detect = 1 when the third bit of a matching sequence arrives.

Moore states:

  • S0 (IDLE): no progress; output 0
  • S1 (GOT_1): last bit was 1; output 0
  • S2 (GOT_10): last two bits were 10; output 0
  • S3 (GOT_101): sequence complete; output 1
Current stateInput = 0Input = 1Output
S0 IDLE→ S0→ S10
S1 GOT_1→ S0→ S10
S2 GOT_10→ S0→ S30
S3 GOT_101→ S0→ S11
Overlap handling: From S3, input 1 goes to S1 (not S0) because the last '1' of "101" can start a new "101" sequence — this is overlapping detection.

With 2-bit binary encoding (S0=00, S1=01, S2=10, S3=11), derive next-state and output equations from the table, then implement with two D flip-flops and combinational logic.

           └──────────────────────────────┘                  │
                              Clock ─────────────────────────┘
  • Moore: output logic takes only State S
  • Mealy: output logic takes State S and Inputs X
  • Both models are equivalent in expressive power — any Mealy FSM can be converted to Moore (possibly with more states) and vice versa
Common mistake: Incomplete state tables — if an input combination is unspecified, synthesis tools may produce unpredictable hardware. Always define all transitions or explicit default behaviour.
